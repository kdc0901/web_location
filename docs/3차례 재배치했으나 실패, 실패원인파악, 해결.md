## 실패 원인 파악

### 1) push는 성공했는가? — **예**

터미널로 확인했을 때:

```bash
# 저장소 (push된 main)
curl -sL "https://raw.githubusercontent.com/kdc0901/web_location/main/js/data.js" \
  | grep "group: 1, seat: 1"
# → { name: "우승희", ... }

# 실제 사이트
curl -sL "https://kdc0901.github.io/web_location/js/data.js" \
  | grep "group: 1, seat: 1"
# → { name: "이기옥", ... }
```

**코드 저장소 ≠ 사용자가 보는 사이트** 였습니다.

### 2) `?v=6` / `?v=7` 문제인가? — **아니오 (주원인 아님)**

`?v=` 는 브라우저·캐시 우회용입니다.  
**서버에 올라간 `data.js` 파일 자체**가 예전이면, URL에 아무 번호를 붙여도 **이기옥**입니다.

### 3) 휴대폰 캐시만 문제인가? — **아니오**

`github.io`에서 받은 파일의 `last-modified`가 **5월 17일 06:54** (마지막 **built** 시각)로 고정되어 있었습니다.  
→ **CDN·Pages가 v5 파일만 서빙** 중이었습니다.

### 4) 진짜 원인 — **GitHub Pages 빌드 연속 실패**

| 커밋 | 메시지 | Pages `status` | `error` |
|------|--------|----------------|---------|
| `890d16d` | v5 복귀 | **built** | — |
| `72a03f2` | v6 명단 + docs 추가 | **errored** | Page build failed. |
| `ec0304a` | 배포 재시도 | **errored** | Page build failed. |
| `4505362` | v7 명단 | **errored** | Page build failed. |
| `5718445` | 배포 재시도 | **errored** | Page build failed. |
| `0b4d041` | **.nojekyll** | **built** | — |

**v6 커밋 이후 한 번도 `built`가 없어서**, 사이트는 **5/17 v5(이기옥)** 에 멈춰 있었습니다.

### 5) 빌드가 왜 깨졌나? — **Jekyll이 정적 사이트가 아닌 것처럼 처리**

GitHub Pages(legacy)는 기본으로 **Jekyll** 빌드를 시도합니다.

`main` 브랜치에 아래가 같이 있으면 빌드가 실패하기 쉽습니다.

| 요인 | 이 프로젝트 예 |
|------|----------------|
| `docs/` 대량 md | 6_1)~7_6) 한글·괄호 파일명 |
| 이미지 | `docs/전체배치도.jpeg` |
| Node/Electron 흔적 | `package.json`, `package-lock.json`, `electron/` 등 |
| **`.nojekyll` 없음** | Jekyll이 전부 “사이트 소스”로 처리 |

실제 앱은 **루트의 `index.html` + `js/` + `css/`** 만 있으면 되는데, Jekyll이 `docs/` 등을 건드리다 **`Page build failed`** 가 난 것으로 보는 것이 타당합니다.

## 해결 방법

### 적용한 수정

프로젝트 **루트**에 빈 파일 추가:

```
.nojekyll
```

| 효과 |
|------|
| GitHub Pages가 **Jekyll 빌드를 건너뜀** |
| `index.html`, `js/data.js` 등을 **그대로** 배포 |

### 커밋·push

```bash
cd "/Volumes/Exe, Todo(md,book,app)_2T mac/_location"

git add .nojekyll

git commit -m "$(cat <<'EOF'
fix: GitHub Pages 빌드 실패 방지 (.nojekyll 추가)

EOF
)"

git push origin main
```

실제 해결 커밋: **`0b4d041`**

### 해결 확인

```bash
gh api repos/kdc0901/web_location/pages/builds/latest \
  --jq '{status, commit: .commit[0:7], error: .error.message}'
# → status: "built"

curl -sL "https://kdc0901.github.io/web_location/js/data.js" \
  | grep "group: 1, seat: 1"
# → { name: "우승희", group: 1, seat: 1 },
```

브라우저:

```
https://kdc0901.github.io/web_location/?v=7
```

**(가) 두레 1번 = 우승희** 이면 정상.

## 증상별 판단 표 (다음에 비슷할 때)

| 보이는 현상 | 원인 후보 | 확인 방법 |
|-------------|-----------|-----------|
| push 했는데 사이트만 예전 명단 | **Pages errored** | Deployments / `gh api .../builds/latest` |
| `?v=` 올려도 동일 | 서버 파일 미갱신 | raw vs github.io `data.js` curl |
| 로컬·GitHub는 새 이름 | push OK, **배포 실패** | 위와 동일 |
| 빌드 `building`만 오래 지속 | 멈춤 또는 곧 errored | 10분 후 재확인 |
| v6 이후 docs 많이 추가함 | Jekyll 실패 가능 | **`.nojekyll`** 유지 |

## 앞으로 명단만 다시 돌릴 때

1. `js/data.js` · `index.html` · `sw.js` 수정 (기존 문서 `8_...md` 참고)
2. `git push`
3. **반드시** 빌드 `built` 확인 (`배포 빌드가 되었는지 아닌지 확인.md`)
4. **raw vs github.io** `data.js` 내용 같을 때까지 기다리기
5. `?v=번호` 로 휴대폰 확인

**`.nojekyll`은 삭제하지 마세요.** 삭제하면 같은 빌드 실패가 다시 날 수 있습니다.

## 교훈 (3줄)

1. **`git push` 성공 ≠ 사이트 반영** — Pages **`built`** 를 꼭 확인한다.  
2. **`?v=` 번호만 올려서는** 서버에 예전 `data.js`가 있으면 명단이 안 바뀐다.  
3. 정적 HTML 사이트 + `docs/`·`package.json`이 같이 있으면 **`.nojekyll`** 을 루트에 둔다.
